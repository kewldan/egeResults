# Code review — ege-notifier

Ревью от 2026-06-01. Найденные баги и предложения по улучшению, по убыванию важности.
Статус: все 4 бага исправлены; из улучшений применены троттлинг рассылки,

## Баги

### 1. Фамилия с пробелом превращается в дефис → ломает поиск на сайте ✅ исправлено
**Файл:** `src/ege_notifier/bot/validators.py:12`

```python
return "-".join(part.capitalize() for part in v.replace(" ", "-").split("-") if part)
```

Двойная фамилия «Салтыков Щедрин» станет «Салтыков-Щедрин». Сайт ege.spb.ru
сопоставляет фамилию точно (вместе с паспортом) → запрос вернёт пусто, и
пользователь будет думать, что результатов нет. Регистр нормализовать правильно
(сайт ждёт формат «Тенишев»), но разделитель менять нельзя.

**Фикс:**
```python
return re.sub(r"[А-Яа-яЁё]+", lambda m: m.group().capitalize(), " ".join(v.split()))
```
Капитализирует каждое слово, сохраняя и пробел, и дефис.

### 2. Новый подписчик уже отслеживаемого ученика не видит существующие результаты ✅ исправлено
**Файлы:** `src/ege_notifier/bot/handlers/add_student.py` (confirm_add) +
`src/ege_notifier/services/results.py` (`check_student`).

`check_student` шлёт только *изменения* относительно сохранённого снимка. Для
нового ученика снимок пуст → всё уйдёт как «новое». Но если на ученика уже
кто-то подписан и его баллы давно в базе, второй подписчик после подписки
**не получит ничего** (diff пуст), хотя результаты есть.

**Фикс:** если `subscribe()` вернул существующего ученика с непустым
`student.results`, отправить новому подписчику снимок текущих результатов
независимо от diff (отдельный форматтер «текущие результаты»).

### 3. Гонка при создании ученика/пользователя → падение хендлера ✅ исправлено
**Файлы:** `src/ege_notifier/services/subscriptions.py:52-63` (`get_or_create_student`),
там же `upsert_user`.

`find_one` → `insert()` без защиты от гонки: два пользователя, одновременно
подписывающиеся на один паспорт (или двойной /start), оба получат `None`, оба
вставят → уникальный индекс `identity_hash` / `telegram_id` бросит
`DuplicateKeyError`, хендлер упадёт. Редко, но реально.

**Фикс:** поймать `pymongo.errors.DuplicateKeyError` и перечитать запись, либо
использовать `find_one_and_update(..., upsert=True)`.

### 4. Fire-and-forget задача на старте ✅ исправлено
**Файл:** `src/ege_notifier/__main__.py`

```python
asyncio.create_task(run_check_cycle(results, notifier))
```

Ссылка на задачу не сохранена (риск преждевременного GC), исключения молча
проглатываются.

**Фикс:** сохранить ссылку и навесить `add_done_callback` с логированием ошибок.

## Улучшения (не баги)

- ✅ **Хрупкий ключ предмета** (частично). `_normalize_subject` теперь
  отбрасывает пунктуацию, поэтому «Математика профильная» и «Математика
  (профильная)» дают один ключ. Полноценный словарь канонических кодов —
  отдельная задача (нужен список всех предметов), отложено.
- ⏳ **Не отличает «опечатку / не найден» от «результатов пока нет».** Обе ситуации
  дают пустой список. Детектить наличие формы/сообщения об ошибке на странице и
  подсказывать пользователю проверить данные. (Отложено — требует анализа реальной
  страницы.)
- ✅ **Гонка планового и ручного `check`.** Добавлена блокировка по `student.id`
  в `ResultsService` + перечитывание актуального снимка под блокировкой, так что
  параллельные проверки не дают lost update / дубль-уведомление.
- ⏳ **`MemoryStorage` теряет FSM при рестарте.** Для прод — Redis/персистентное
  хранилище. (Отложено — инфраструктурное.)
- ✅ **`drop_pending_updates` на старте** включён в `dp.start_polling`.
- ✅ **Троттлинг рассылки.** `Notifier.broadcast` делает паузу `broadcast_delay`
  (по умолчанию 0.05 с) между сообщениями.
- ✅ **`check_all` грузит всех учеников + N+1.** Теперь подписки берутся одним
  запросом (`subscribers_by_student`), грузятся только ученики с подписчиками.
- ✅ **tz-aware vs naive datetime.** Клиент создаётся с `tz_aware=True`.
- ⏳ **Мёртвые ветки.** После реализации `ege_spb` `NotImplementedError` /
  `texts.PROVIDER_NOT_READY` больше не срабатывают — оставлены как защита на
  будущее (для следующих провайдеров), удаление отложено.

## Ревью №2 (по итогам первых правок) — все исправлены

Повторное ревью самих правок нашло ещё 5 проблем; все устранены.

### 1. Смена `_normalize_subject` рвала связь со старым снимком ✅ исправлено
Изменение нормализации ключа предмета (отбрасывание пунктуации) не совпадало с
уже сохранёнными в БД ключами → первая же проверка после деплоя дала бы ложное
«🆕» по каждому такому предмету и дубль записи. Нормализация вынесена в
`utils.normalize_subject`, а `services/diff.py` теперь нормализует **обе**
стороны сопоставления и хранит канонический ключ — старые ключи лениво
переезжают на новый без дублей и ложных уведомлений (тесты в `test_diff.py`).

### 2. Ручная проверка при подписке уведомляла только нового подписчика ✅ исправлено
`confirm_add` записывал свежий снимок в БД, но слал изменения лишь
`from_user` — остальные подписчики ученика навсегда теряли это уведомление
(плановая проверка diff уже не видела). Теперь рассылается всем подписчикам
ученика (`subscribers_for` + `notifier.broadcast`).

### 3. Воскрешение удалённого ученика в `check_student` ✅ исправлено
`unsubscribe` (без блокировки) мог удалить ученика во время `fetch`, а
`student.save()` (upsert) воскрешал его с PII. Запись переведена на
`replace()` (без upsert) с перехватом `DocumentNotFound` — удалённый ученик не
воскрешается и не уведомляется (`test_check_student_does_not_resurrect…`).

### 4. Неограниченный рост `_locks` ✅ исправлено
Словарь блокировок копил по одному `asyncio.Lock` на каждого когда-либо
проверенного ученика. Заменён на `weakref.WeakValueDictionary` —
запись живёт только пока блокировку кто-то держит (`test_results.py`).

### 5. `format_current_results` на пустом списке давал «голый» заголовок ✅ исправлено
Добавлена защита: при пустом снимке возвращается «Результатов пока нет».
